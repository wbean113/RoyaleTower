# 高级语言程序设计大作业实验报告

## 一. 作业题目

RoyaleTower（皇家塔防）——基于Qt的2D图形化塔防游戏。

## 二. 开发软件

Qt Creator 10.0.2 + Qt 6.11.0（MinGW 64-bit编译器），C++17标准，qmake构建系统。

## 三. 课题要求

1) 面向对象——使用C++类的继承、多态、封装特性组织游戏逻辑。
2) 图形化交互——基于Qt Graphics View框架实现2D精灵渲染与UI面板。
3) 核心游戏逻辑——包含关卡系统、敌人波次、塔防放置、法术释放、胜负判定等完整塔防玩法。
4) 功能验证——通过单元测试覆盖各模块功能。

## 四. 主要流程

### 1．整体流程

**实现思路：**

项目采用Qt Graphics View框架构建游戏场景。核心类继承关系如下：

- **GameScene** 继承自 `QGraphicsScene`，管理所有游戏对象（敌人、防御塔、法术特效、HUD）、关卡路线、敌人生成波次、塔防放置网格、游戏胜负判定与暂停/恢复等全部核心逻辑。
- **Enemy** 继承自 `QGraphicsEllipseItem`，封装敌人属性（类型/HP/速度/状态效果）与路径跟随移动。
- **Tower** 多重继承自 `QObject + QGraphicsRectItem`，封装防御塔的攻击冷却、射程内目标搜索与弹道特效。
- **CardItem** 继承自 `QGraphicsObject`（即 QObject + QGraphicsItem），作为卡牌面板中单张卡牌的图形项，发射信号实现界面与逻辑解耦。
- **GameView** 继承自 `QGraphicsView`，封装场景的视图缩放与显示控制。

**游戏主循环：**

使用5个定时器驱动游戏运行：

| 定时器 | 间隔 | 职责 |
|--------|------|------|
| resourceTimer | 2000ms | 每2秒增加圣水资源 |
| spawnTimer | 2500ms | 每2.5秒生成一波敌人 |
| gameLoopTimer | 50ms | 游戏逻辑更新（敌人移动、塔攻击、技能效果） |
| cardCooldownTimer | 200ms | 卡牌冷却倒计时更新 |
| spellEffectTimer | 100ms | 法术特效过期清理 |

根据用户操作类型，使用基类指针虚函数引用调用子类实现，达到多态效果。通过Qt信号槽机制实现界面与逻辑的解耦。

**瓦片地图系统：**

游戏场景采用80×80像素瓦片地图架构，共20列×9行（180个瓦片格）。每个关卡由 `int map[9][20]` 二维数组定义，包含13种瓦片类型：

- 0：草地（可放置防御塔）
- 1：垂直道路
- 2：水平道路
- 3~6：四种转弯道路
- 7：十字路口
- 8：直线道路
- 9~13：装饰瓦片（每局开始时从-1占位符随机替换）

道路瓦片（类型1~8）的方向编码隐式定义了敌人路径拓扑。敌人沿道路瓦片逐格推进，根据方向编码推算下一个前进方向。

**关卡设计：**

三个关卡各具不同的路线布局：
- **关卡1**：单条水平直线路径（row 4全行道路），最简单。
- **关卡2**：回字型单路循环，围绕外围矩形环行走，中等难度。
- **关卡3**：Y字型双路汇合路线，两条路径在中下方汇合后直下，最高难度。

每个关卡的初始资源、每波敌人数、出生点数量与位置均为独立配置。随着波次推进，高血量敌人（Tank、Boss、king1、king2）的出现概率逐渐增大，实现难度递增。

**HUD系统：**

左侧状态面板（半透明深色背景）实时显示：圣水资源（金色）、关卡编号、当前波次、总击杀数、存活敌人数量、损失血量。右侧卡牌面板横向排列7张卡牌（箭塔、炮塔、冰塔、魔法塔、火球术、冰冻术、闪电术），支持滚轮滚动浏览，每张卡牌有固定费用和5000ms冷却时间。

**游戏流程：**

开始界面 → 关卡选择界面（3个关卡按钮 + 难度选择） → 游戏主场景 → 波次推进 → 胜利/失败判定 → 返回主菜单。

### 2．算法或公式

以下 `(col, row)` 代表网格坐标，`(x, y)` 代表场景像素坐标，`d` 代表两点间距离。

#### （1）网格坐标转换

将网格逻辑坐标转换为场景像素中心坐标：

```
x = col * GRID_SIZE + GRID_SIZE / 2
y = row * GRID_SIZE + GRID_SIZE / 2
```

其中 `GRID_SIZE = 80` 为每个瓦片的像素尺寸。反向计算用于 `canPlaceAnything()` 和 `isPathTile()` 中将场景像素坐标还原为网格坐标 `(col, row)`。

#### （2）防御塔索敌算法

防御塔基于曼哈顿距离搜索射程内最近的敌人：

```
for each enemy in enemies:
    if enemy.isDead(): continue
    dx = enemy.x - tower.x
    dy = enemy.y - tower.y
    d = |dx| + |dy|  // 曼哈顿距离
    if d <= tower.range and d < minDist:
        minDist = d
        target = enemy
```

遍历所有存活敌人，取曼哈顿距离最小且位于射程内的敌人作为攻击目标。优先攻击距离最近的敌人。

#### （3）敌人路径跟随算法

敌人在瓦片地图上沿道路方向编码拓扑逐格移动。每50ms（gameLoopTimer间隔）更新位置：

```
current_index = enemy.pathIndex
target_point = path[current_index]
direction = (target_point - enemy.pos).normalized()
if manhattanLength(enemy.pos, target_point) < enemy.speed:
    enemy.pos = target_point
    enemy.pathIndex = current_index + 1
else:
    enemy.pos += direction * enemy.speed
```

当敌人距离当前目标路点小于 `speed` 像素时，自动吸附到该路点并切换到下一个路点。被冰冻（frozenRemaining > 0）时跳过移动更新。

#### （4）法术伤害计算

三种法术的伤害逻辑各不相同：

**火球术（范围伤害，60像素半径）：**
```
for each enemy in enemies:
    d = sqrt((spell.x - enemy.x)² + (spell.y - enemy.y)²)
    if d <= 60:
        enemy.hp -= 15 * (1 - d / 60)
```
伤害随距离线性衰减，中心伤害15点，边缘伤害趋近于0。

**冰冻术（全屏控制）：**
对所有存活敌人施加 `applyFreeze(2000)`，冻结2秒（速度降为0，不能移动）。

**闪电术（精确打击）：**
对距离终点最近的3个存活敌人各造成25点伤害。

**测试结果**

测试过程中发现两个主要问题：

**Bug 1 —— SpellEffectItem 悬空指针崩溃。** 法术特效类 `SpellEffectItem` 在构造函数中使用 `QTimer::singleShot(lifetimeMs, callback)` 设置定时器，在特效播放完毕后自动从场景移除自身。同时，游戏循环通过 `spellEffectTimer`（原间隔1000ms）定期调用 `clearSpellEffects()` 遍历并清理特效列表。两种清理机制存在竞态条件——若 `QTimer::singleShot` 回调在 `clearSpellEffects()` 持有指针期间触发，将导致访问已释放的内存。修复方案：完全移除 `QTimer::singleShot` 调用，在 `SpellEffectItem` 中添加 `m_elapsed` 累计存活时间字段和 `isExpired()` 判定方法，在游戏主循环（50ms间隔）中同步累加存活时间，在 `spellEffectTimer`（间隔缩短为100ms）中基于 `isExpired()` 统一清理过期特效。重构后法术相关的随机崩溃彻底消除。

**Bug 2 —— 路径方向编码歧义导致敌人路线错误。** 关卡2回字型和关卡3双路汇合路线中，部分转弯瓦片的方向编码与路径流向不匹配。例如关卡2中右上角转弯瓦片错误标记为类型3（从上方进入向右方转出），但实际敌人应从右侧进入向下方转出，应使用类型6。关卡3十字路口处未正确使用类型7（十字路口）标记所有出入口方向。通过逐个关卡追踪敌人完整行走路径，修正了关卡2四个转角的方向编码和关卡3汇合点的类型值，测试通过。

## 六. 收获

### 1．SpellEffectItem 内存安全重构

```cpp
// 原代码：构造函数中使用异步回调
SpellEffectItem::SpellEffectItem(int lifetimeMs, QGraphicsScene *scene)
{
    scene->addItem(this);
    QTimer::singleShot(lifetimeMs, [this, scene]() {
        scene->removeItem(this);
        delete this;  // 异步回调中释放自身
    });
}

// 在游戏循环中另有 clearSpellEffects() 同步清理
void GameScene::clearSpellEffects()
{
    for (auto effect : spellEffects) {
        removeItem(effect);
        delete effect;  // 可能与 singleShot 冲突
    }
    spellEffects.clear();
}
```

以上两段代码存在**竞态条件**：`QTimer::singleShot` 的回调在Qt事件循环中异步执行，而 `clearSpellEffects()` 在主循环中同步执行。如果 `clearSpellEffects()` 先执行了 `delete effect`，随后 `singleShot` 回调触发访问已释放内存，导致程序崩溃。反之亦然。

**修复方案**：移除异步回调，改为在游戏主循环中同步管理生命周期。

```cpp
// 修复后：同步管理生命周期
class SpellEffectItem {
    int m_elapsed = 0;       // 累计存活时间
    int m_lifetime;           // 特效总生命周期
public:
    bool isExpired() const { return m_elapsed >= m_lifetime; }
};

// 在 updateGame() 中每50ms累加存活时间
effect->m_elapsed += 50;

// clearSpellEffects() 基于 isExpired() 统一清理
void GameScene::clearSpellEffects()
{
    for (int i = spellEffects.size() - 1; i >= 0; --i) {
        if (spellEffects[i]->isExpired()) {
            removeItem(spellEffects[i]);
            delete spellEffects[i];
            spellEffects.removeAt(i);
        }
    }
}
```

**收获**：异步回调（如 `QTimer::singleShot`）与同步遍历（如 `clearSpellEffects()`）的混用极易引发竞态条件，导致难以复现的随机崩溃。应将对象的生命周期管理统一到同一执行路径中。Qt中 `QTimer::singleShot` 配合 `delete this` 的模式看似便捷，但在复杂系统中应与同步清理逻辑二选一，不可并存。

### 2．路径方向编码调试

瓦片地图系统的道路方向编码（1~8）在路径拓扑中有明确的方向含义。关卡2回字型和关卡3双路汇合路线中，部分转弯瓦片的方向编码与周围道路瓦片的方向拓扑不连贯，导致敌人在转弯处停止移动或走错方向。

```cpp
// 关卡2地图数据中，右上角转弯瓦片原编码：
{ ..., 3, ... }  // 类型3：从上方→向右方
// 实际路径流向：从水平（右→左）→垂直（上→下）
// 应使用类型6：从右方→向下方
{ ..., 6, ... }
```

通过逐格追踪敌人完整行走路径、在每个转弯处打印当前瓦片类型与敌人实际移动方向，快速定位到编码错误的具体位置。修改为正确的方向编码后，所有关卡敌人路径测试通过。

**收获**：基于方向编码的路径拓扑系统，编码值的选择必须严格匹配实际路径流向。调试时采用"逐格追踪+关键点日志"的策略，比全局搜索debug更高效。此外，瓦片编码应有明确的文档对照表（如类型3=上→右，类型4=左→下等），仅靠数字记忆容易出错。
